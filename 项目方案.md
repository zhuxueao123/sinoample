# 自动售货机海外 B2B 网站项目方案

## 1. 项目定位

本项目是面向海外客户的自动售货机 B2B 官网，主要目标不是在线购买，而是展示产品、建立信任、获取销售线索，并引导客户提交联系方式，由销售团队线下沟通、报价和完成下单。

网站界面和内容使用英文，整体风格应专业、清晰、可信，符合海外 B2B 客户的浏览习惯。

## 2. 目标客户

主要客户群体包括：

- 海外经销商和代理商
- 自动售货机运营商
- 办公楼、学校、医院、酒店、机场等场景采购方
- 零售、连锁、加盟类客户
- 有 OEM / ODM 定制需求的企业客户

## 3. 网站核心目标

- 展示自动售货机产品线和产品参数
- 说明不同商业场景下的解决方案
- 强化公司实力、工厂能力、质量控制和售后支持
- 获取客户询盘信息
- 让销售团队根据客户需求进行线下报价和跟进

## 4. 顶部导航规划

## 9. 技术选型与部署方案

### 9.1 初步技术栈

建议采用以下技术栈：

- 前端网站：Next.js
- CMS 后台：Strapi
- 数据库：PostgreSQL
- 媒体资源存储：Cloudflare R2
- 前端部署：Cloudflare Pages 或 Cloudflare Workers
- 域名、CDN、缓存、安全防护：Cloudflare
- 询盘通知：邮件通知为基础，后续可扩展到 CRM、WhatsApp 或企业内部通知工具

### 9.2 Cloudflare 部署边界

Cloudflare 非常适合部署前端网站、缓存静态资源、做全球 CDN、安全防护和表单接口。

但 Strapi 是完整的 Node.js CMS 服务，需要长期运行的 Node 进程、数据库连接、后台管理界面和媒体上传能力。因此不建议把 Strapi 直接部署为 Cloudflare Pages 或普通 Cloudflare Workers 应用。

建议采用分离部署：

- `www.sinoample.shop`：Next.js 前端，部署到 Cloudflare。
- `admin.sinoample.shop`：Strapi CMS，部署到公司内部服务器，通过公网域名访问。
- `media.sinoample.shop`：产品图片、博客封面、资料文件等媒体资源，存储在 Cloudflare R2。

### 9.3 推荐架构

推荐架构如下：

1. 海外客户访问 Next.js 前端网站。
2. 前端从 Cloudflare D1 读取公开展示内容。
3. Strapi 作为公司内部 CMS，用于维护产品、文章、方案、FAQ 等内容。
4. Strapi 内容发布、更新或删除后，通过同步服务把公开内容同步到 Cloudflare D1。
5. 图片、PDF、手册等资源存储在 Cloudflare R2，并通过 Cloudflare CDN 分发。
6. 客户提交 Contact Us 或 Get a Quote 表单。
7. 表单数据通过 Cloudflare Worker 写入 D1，并触发邮件通知销售团队。
8. 销售团队在 Strapi 后台的自定义管理页面中查询 D1 里的询盘记录，并线下跟进。

### 9.4 内容管理范围

Strapi 后台建议维护以下内容：

- Products：产品信息、规格、图片、文件下载、SEO 信息
- Product Categories：产品分类
- Solutions：应用场景和解决方案
- Blog / Insights：博客文章、分类、标签、作者、SEO 信息
- Support：FAQ、下载资料、服务说明、保修政策
- Company Content：公司介绍、证书、工厂能力、全球市场
- Inquiries：通过自定义后台页面查询 D1 中的客户询盘、来源页面、感兴趣产品、跟进状态

说明：询盘数据的主数据不放在 Strapi 自身数据库中，而是放在 Cloudflare D1。Strapi 后台只作为销售和管理员查看、处理询盘的入口。

### 9.5 前端渲染策略

考虑到这是 B2B 展示型网站，优先建议使用静态生成和缓存策略：

- 产品列表页、产品详情页、解决方案页、博客页优先静态生成。
- 内容更新后，通过 Strapi Webhook 触发 Cloudflare 重新构建或清理缓存。
- Contact Us 和 Get a Quote 表单使用动态接口提交。
- 对 SEO 重要页面生成独立 URL、title、description、Open Graph 信息和结构化数据。

URL 规则：

- 产品详情页：`/products/{category-slug}/{product-slug}`
- 博客详情页：`/blog/{slug}`

这样可以兼顾海外访问速度、SEO 和维护便利性。

### 9.6 注意事项

- Strapi 生产环境不要使用本地 SQLite，建议使用 PostgreSQL。
- Strapi 上传文件不要保存在本地磁盘，建议接 Cloudflare R2 或其它 S3 兼容对象存储。
- Next.js 需要根据 Cloudflare 的支持方式选择静态导出、Cloudflare Pages 或 Workers 运行方式。
- 如果不需要复杂 SSR，优先采用更静态化的方案，部署更稳定，速度也更好。

## 10. D1 前台读库与内容同步方案

### 10.1 设计目标

由于 Strapi 计划部署在公司内部服务器上，不能保证高可用，因此前台网站不能实时依赖 Strapi 在线状态。

本项目采用以下思路：

- Strapi 负责内容维护。
- Cloudflare D1 负责前台公开内容读取。
- Cloudflare R2 负责公开媒体资源存储。
- Cloudflare Worker 负责内容同步、前台接口和询盘接口。

这样即使公司内部 Strapi 暂时不可访问，海外客户仍然可以正常访问网站、浏览产品和提交询盘。

### 10.2 数据流

内容同步流程：

1. 管理员在 Strapi 后台维护产品、博客、解决方案等内容。
2. 内容发布、更新或删除时，Strapi Webhook 调用 Cloudflare Worker。
3. Worker 拉取 Strapi API 中的已发布内容。
4. Worker 将内容转换成前台需要的结构。
5. Worker upsert 到 Cloudflare D1。
6. 前台网站从 D1 读取内容。

询盘流程：

1. 客户在前台 Contact Us 或 Get a Quote 表单中提交信息。
2. Cloudflare Worker 校验表单、做人机验证和频率限制。
3. Worker 将询盘写入 D1。
4. Worker 发送邮件通知销售团队。
5. 销售团队在 Strapi 后台自定义页面中查看和处理询盘。

### 10.3 D1 不复制 Strapi 全库

D1 只保存前台展示和询盘所需的数据，不复制 Strapi 的内部数据库结构。

不应同步到 D1 的内容：

- Strapi 后台用户
- 权限和角色
- 插件表
- 草稿内容
- 内部备注
- 未发布内容
- Strapi 系统表

建议同步到 D1 的公开内容：

- products
- product_categories
- solutions
- blog_posts
- blog_categories
- faqs
- downloads
- site_settings

建议字段：

- id
- source_page

- product_name

- name
- company
- email
- phone
- whatsapp
- country
- quantity
- message
- status
- assigned_to
- email_sent_at
- email_status
- email_error
- zoho_message_id
- sales_note
- created_at
- updated_at

询盘状态建议：

- new
- contacted
- quoted
- won
- lost
- spam

### 10.5 询盘 API

Cloudflare Worker 建议提供以下接口：

- `POST /api/inquiries`：前台提交询盘。
- `GET /api/admin/inquiries`：后台查询询盘列表。
- `GET /api/admin/inquiries/:id`：后台查询询盘详情。
- `PATCH /api/admin/inquiries/:id`：后台更新询盘状态或销售备注。

权限要求：

- `POST /api/inquiries` 是公开接口，但必须做字段校验、Turnstile、防刷和频率限制。
- `GET /api/admin/inquiries`、`GET /api/admin/inquiries/:id`、`PATCH /api/admin/inquiries/:id` 是后台接口，必须鉴权。

### 10.6 Strapi 后台查询 D1 询盘

推荐采用 Strapi 后端代理方式，而不是让 Strapi Admin 前端直接暴露管理 API Token。

后台查询链路：

1. Strapi Admin 自定义页面发请求到 Strapi 自定义 API。
2. Strapi 自定义 API 使用服务器端密钥请求 Cloudflare Worker Admin API。
3. Worker Admin API 查询或更新 D1。
4. 查询结果返回 Strapi Admin 页面展示。

这样浏览器端不会暴露 Worker 管理接口密钥，也可以复用 Strapi 后台登录态和权限控制。

询盘后台需要支持：

- 查看询盘列表
- 查看询盘详情
- 按国家 / 地区分配销售员工
- 修改询盘状态
- 添加销售备注
- 分配销售员工
- 导出 CSV
- 手动补发邮件通知

### 10.7 同步保障

为降低同步失败风险，建议同时使用两种机制：

- Webhook 增量同步：内容发布、更新、删除时立即同步。
- 定时全量校验：每小时或每天同步一次已发布内容，修复漏同步或失败数据。

同步时需要处理：

- 发布状态
- 删除状态
- slug 变化
- 产品分类关系
- 产品和解决方案的关联
- 图片和下载文件 URL
- SEO 字段

## 11. 邮件服务方案

### 11.1 邮件使用场景

本项目涉及两类邮件需求：

1. 公司对外邮箱，例如 `sales@example.com`、`info@example.com`、`support@example.com`。
2. 系统通知邮件，例如客户提交询盘后，系统通知销售团队。

### 11.2 Cloudflare 能力边界

Cloudflare 可以管理域名 DNS，并提供 Email Routing、Email Workers 和 Email Service 相关能力。

需要区分：

- Email Routing：主要用于收信转发，把 `sales@example.com` 等地址收到的邮件转发到指定邮箱。
- Email Workers：可以处理进入 Cloudflare 的邮件，也可以配合 Worker 做自动化逻辑。
- Email Sending / Email Service：可从 Worker 或 API 发送事务性邮件，用于系统通知类场景。

但如果要做完整的公司邮箱，例如销售人员用 `sales@example.com` 正常收发、回复客户、管理邮箱、IMAP/SMTP、网页版邮箱、移动端邮箱客户端、日历和联系人等，仍建议使用专业邮箱服务，例如 Google Workspace、Microsoft 365、Zoho Mail 或企业邮箱服务。

截至当前方案阶段，Cloudflare Email Service 更适合应用邮件能力，例如事务邮件发送、邮件路由、Worker 处理入站邮件、通知和自动回复；它不是传统意义上的完整企业邮箱套件。

### 11.3 推荐邮件架构

建议采用：

- 公司业务邮箱：使用 Zoho Mail。
- 域名 DNS：继续由 Cloudflare 管理。
- 系统询盘通知：Cloudflare Worker 调用 Zoho Mail API 发送邮件。
- 系统发件地址：使用 `noreply@sinoample.shop`。
- 收信别名：根据 Zoho Mail 方案配置，例如 `sales@sinoample.shop`、`info@sinoample.shop`、`support@sinoample.shop`。

### 11.4 询盘通知流程

询盘通知建议流程：

1. 客户提交 Contact Us 或 Get a Quote 表单。
2. Cloudflare Worker 校验表单。
3. Worker 将询盘写入 D1。
4. Worker 根据客户国家 / 地区匹配对应销售员工。
5. Worker 通过 Zoho Mail API 使用 `noreply@sinoample.shop` 发送通知邮件给对应销售员工。
6. 邮件中包含客户信息、来源页面、感兴趣产品、国家、预计数量和 D1 询盘 ID。
7. 销售人员进入 Strapi 后台查看完整询盘和跟进状态。

### 11.5 邮件配置要点

无论使用 Cloudflare 发信还是第三方事务邮件服务，都需要配置正确的 DNS 记录，以提高送达率：

- SPF
- DKIM
- DMARC
- MX
- Return-Path / Bounce 相关配置，视邮件服务商要求而定

如果使用 Cloudflare Email Routing 收信，需要根据 Cloudflare 提示配置 MX 记录。

如果使用第三方邮箱或事务邮件服务，需要按对应服务商要求配置 MX、SPF、DKIM、DMARC，Cloudflare 只负责托管这些 DNS 记录。

### 11.6 初步建议

第一阶段建议：

- 使用 Cloudflare 管理域名和 DNS。
- 使用 Zoho Mail 作为企业邮箱服务。
- 使用 Cloudflare Worker 处理询盘接口、写入 D1，并通过 Zoho Mail API 发送销售通知邮件。
- 如果 Zoho API 发送失败，需要在 D1 中记录失败状态，并支持后台手动补发。

这样前台询盘链路仍然不依赖公司内部 Strapi 服务器，且邮件能力可以按业务增长逐步升级。

### 11.7 企业邮箱选型判断

企业邮箱选型按以下规则判断：

- 如果 Cloudflare 后续提供完整邮箱套件，包括独立邮箱账号、网页邮箱、IMAP/SMTP 或等效客户端协议、移动端支持、管理员用户管理、邮箱容量、别名、群组、日历和联系人，则可以优先考虑 Cloudflare。
- 如果 Cloudflare 只提供 Email Routing、Email Workers 和 Email Sending，则公司邮箱仍使用 Google Workspace、Microsoft 365、Zoho Mail 或其它成熟企业邮箱。
- 询盘通知、自动回复和系统邮件可以继续用 Cloudflare Email Service，因为这类邮件属于应用事务邮件，不要求销售人员在邮箱客户端里直接管理发信服务。

本项目第一阶段已确定使用 Zoho Mail 作为企业邮箱，并通过 Zoho Mail API 发送询盘通知邮件，因此 Cloudflare Email Service 暂不作为主发信方案。

## 12. 已确认的业务决策

### 12.1 销售分配

询盘按客户国家 / 地区分配给不同销售员工。

需要在后台维护国家 / 地区与销售员工邮箱的映射关系。无法匹配时，发送到默认销售邮箱。

### 12.2 询盘管理功能

Strapi 后台自定义询盘页面需要支持：

- 查看询盘列表
- 查看询盘详情
- 修改状态
- 添加销售备注
- 分配销售员工
- 按国家 / 地区自动分配销售员工
- 导出 CSV
- 手动补发邮件

### 12.3 多语言

第一阶段只做英文。

多语言内容维护放到第二阶段再考虑。第一阶段不在 CMS 中启用多语言结构，避免增加内容维护复杂度。

### 12.4 SEO URL

第一阶段使用以下 URL：

- 产品详情页：`/products/{category-slug}/{product-slug}`
- 博客详情页：`/blog/{slug}`

### 12.5 邮件

企业邮箱使用 Zoho Mail。

系统通知使用 `noreply@sinoample.shop` 作为发件地址，通过 Zoho Mail API 发送给对应销售员工。

### 12.6 数据备份

第一阶段不单独为 D1 做备份。

说明：D1 中保存的是前台公开内容读库和询盘数据。公开内容可以从 Strapi 重新同步；询盘数据以 D1 为主记录，但当前阶段不做额外备份要求。

### 12.7 Strapi 访问域名

Strapi 部署在公司内部服务器，通过公网域名访问：

- `admin.sinoample.shop`

### 12.8 产品资料下载

第一阶段不提供产品资料公开下载。

产品手册、证书、datasheet 等资料不在前台提供直接下载入口。如客户需要，由销售线下发送。

### 12.9 隐私合规

前台询盘表单需要加入隐私政策勾选项，例如：

`I agree to the Privacy Policy.`

未勾选时不能提交表单。

## 13. Strapi 内容模型规划

### 13.1 设计原则

Strapi 作为内部 CMS，主要服务内容维护人员和管理员。

设计原则：

- 内容维护友好，字段命名清晰。
- 前台展示需要的数据都可维护。
- 只在 Strapi 中维护源内容，不要求前台实时访问 Strapi。
